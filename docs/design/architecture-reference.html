1 Product model

The whole system is one ladder, and every level is optional.

Containment is optional at every level. Tasks can live alone or directly under a project; work items can live outside any project. A work item is a coalition you form when one piece of work deserves a hub — structure is earned, never demanded. One record, many lenses: action items are work items; canvas nodes and chat cards are live references to the one record — change it anywhere, it changes everywhere.

3 Phase loop & status

One loop learned once; the playbook for that kind of work supplies each phase's checklist.

Planunderstand & decide
Executedo the work
Reviewcheck & refine
Doneshipped
  • Machine IDs plan|execute|review|done are fixed; default labels are Plan/Execute/Review/Done; workspaces may rename labels (agent maps them), but never reorder, add, or remove slots.
  • Phase lives on work items only. A project's stage is derived from its items' phase distribution — no dual truth.
  • Status lives on tasks only: To-do In progress Completed. Task and phase vocabularies are deliberately distinct.
  • Health is always derived — at risk blocked are computed, never hand-set, never a column.
  • Board grammar rule: columns = lifecycle; rows/filters = belonging. Never swapped. Every kanban is the same board, scoped.

4 Navigation

Zen-style: the chrome is stable, the content changes.

  • Workspace switcher top, board dock bottom; sidebars are stable per board and never mutate on a content click.
  • Filters live in the content area, never the sidebar. Object navigation is content tabs, not sidebar swaps.
  • Four required states on every screen before merge: empty · loading · error · populated. No screen merges without all four.
  • Keyboard-complete, WCAG AA contrast, dark mode on every screen.

6 Canvas & docs

Our own live components as first-class canvas objects — all MIT libraries.

  • React Flow for both surfaces — the Workboard graph view and the freeform planning canvas; perfect-freehand for ink.
  • TipTap core + y-prosemirror for docs. BlockSuite exits along with patches/ — do not use it in new code.
  • Graph view is a peer of table/kanban/timeline with full action parity: drag an edge = create a dependency; drop into a phase group = change phase. Layout is presentation; gestures are data.
  • Storage rule: canvases never store object content — only element references, positions, grouping. A prior React Flow attempt failed because data lived only in component state; the lesson is the data layer, not the library.

7 Review queue

One pipeline, all producers. Anything generated lands as a proposal.

  • The proposals table is the review-queue sink: states proposed → accepted | rejected | expired.
  • Idempotent on (source_type, source_id, source_candidate_id, target_kind) — one source routinely yields many proposals (a meeting extracts four action items), so each candidate carries a stable id; true retries dedupe, distinct candidates never collide.
  • Every auto-created object carries provenance and enters via a proposal; agents write directly only to objects they were explicitly assigned.

8 Agents & memory

Agents are first-class members; memory is Postgres truth, served tiered.

Agent surfaces

  • Ambient delegation — "Ask agent" everywhere; the thread binds to the object in view. Centralized supervision in the Agent board. Visible attribution always (agent X, on behalf of user Y).
  • Agent API contract: workspace-scoped machine auth (never browser tokens), idempotency keys, attribution, per-principal rate limits, server-side permission checks — one code path.
  • Custom agents are named, permission-scoped workspace members — created conversationally, mentionable in any thread, assignable like a person.

Memory & context economy

  • Truth in Postgres objects, never loose files: product_strategies, playbooks, workspaces.custom_instructions/ai_memory. A deterministic compiler renders a versioned, read-only context bundle; every run records memory_version.
  • Tiered loading: Tier A core preloaded (~1–2k tokens); Tier B reached via a search_memory tool — exactly like search_tools. Memory grows only through proposals.
  • Context economy (4 disciplines): tool schemas searched not preloaded · memory tiered · data processed in code (Code Mode) · large results spilled to a run-scoped indexed store with search_results/read_result, never inlined.
  • Tier B substrate already exists in schema — the knowledge_base (pgvector-1536 HNSW + search_documents()) and collective_intelligence (get_compressed_context()) migrations. Transform = wiring, not migration.
  • RAG hardening (5): hybrid RRF retrieval (tsvector + vector) · halfvec-1024 for new collections · compression_jobs on the jobs seam · TTL-purged run-scoped collections · a golden-question eval loop. All stock pgvector — runs unchanged on Neon.

9 MCP gateway

Hundreds of servers, model-portable tool search.

  • Three meta-tools — search_tools / load_tool / call_tool — with Hermes-Agent BM25 + ≥10%-context auto-mode (MIT, adopted). Router-level search is model-portable; vendor defer_loading is locked-in.
  • Catalog synced from the official MCP Registry (~9,600 servers), three tiers: Verified · Indexed · Custom.
  • Connection scope is user | workspace (workspace = admin-only, badged); remote-transport-first.
  • Connectors are API-key-first (no OAuth apps we own): Meta system-user token · GA service-account JSON · GitHub PAT — direct API → snapshot cache → recharts.

11 Agent execution tiers

Three tiers, escalate by workload — Railway is the named container fallback.

Hono Worker
Light synchronous LLM calls in the API path.
Dynamic Workers
Instant <5ms throwaway isolates: sub-agent fan-out, MCP Code Mode scripts, sandboxing agent-authored code. ~$0.002/unique worker/day.
Containers
Long durable runs with pause-on-approval; scale-to-zero. Also runs the FastAPI meeting-api. Railway = fallback.

Durable spine = the jobs seam + proposals; run resumability persisted in agent.runs (messages + step cursor).

13 Deployment & seams

One codebase, two editions — SaaS multi-tenant pool + single-tenant self-host. Five seams make it possible.

1 · Auth adapter — Clerk in both editions for v1 (self-host brings its own Clerk); the seam stays the permanent escape hatch. Internal IDs ≠ provider IDs.
2 · Realtime transport — partyserver/DO (SaaS) ↔ Hocuspocus-on-Node (self-host). No DO-specific API leaks out.
3 · Storage = S3 API only — R2 ↔ MinIO; previews behind an image-proxy interface (CF transforms ↔ imgproxy).
4 · Vendor clients behind interfaces — LLM, STT, email with env-configured endpoints.
5 · Jobs seam — CF Queues + Cron (SaaS) ↔ pgmq + pg_cron (self-host).

Open-core: open = the entire product runtime; cloud-only = billing/metering, managed connector OAuth, the hosted MCP fleet, multi-tenant ops. TBD(license) — AGPLv3 vs FSL/BUSL vs MIT, decide before public.

14 Cost doctrine

Cloudflare-first for primitives, Postgres for relational truth only, nothing provider-proprietary in the data layer.

  • Cloudflare prices by usage; Supabase Pro is a fixed ~$25/project once free limits hit — so queues, cron, KV, files, realtime live on Cloudflare.
  • Consolidation is itself the goal: every platform removed is an endpoint, a bill, a deploy pipeline, a credentials set, and a failure mode we stop carrying.
  • End-state operational surface: Cloudflare + Neon + Clerk + pay-per-use keys — all usage-priced, zero fixed floors. Every primitive has a named exit through the seams; one-vendor simplicity, no one-way doors.
