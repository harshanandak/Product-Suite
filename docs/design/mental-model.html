The mental model

One skeleton for business work, software work, and daily tasks — for people and agents.

1 · Three words: project → work item → task

The whole system is one ladder. A project is a goal with a date. A work item is one piece of it — a coalition of everything about that piece. A task is one action.

"Release" is never a fourth word — a software release is a project of kind software. Everything in a coalition is a live reference to one record — change it anywhere, it changes everywhere.

Every level is optional. Tasks can live alone, or directly under a project with no work item in between; work items can live outside any project. A work item is just a coalition you form when one piece of work deserves a hub — structure is earned, never demanded.

2 · One phase loop for everything

Four phases, learned once. The loop never changes — the playbook for that kind of work supplies what each phase's checklist contains.

Planunderstand & decide

Plan = design-thinking discover/define · Execute/Review/Done = agile delivery. The methodology lives inside the loop, so nobody has to think about it — and it is never required: a team can run a plain scrum board and ignore phases entirely. Workspaces can rename the labels in their own words (the agent maps them); the four slots and their meanings never change.

3 · Four layers, one question each

Where the ladder sits in the bigger picture. Business and software follow the same skeleton — only the example words change.

Evidence insights · research runs · meeting decisions · feedback · connector data — attached to the items they justify

Work items align to strategy (primary + weighted secondary) — strategy never contains them. Every object links up and down; nothing floats.

4 · Projects: shift between kinds of work, one calendar

Each department lives in its projects; the founder sees all lanes at once. A project can span departments — assign work to a department's queue or straight to a person. Diamonds are milestones; the red line is today.

5 · It grows with you — and agents do the growing

Big ceiling, tiny floor. Start with just tasks; the rest sits quiet at the edge until it earns its place. Nobody configures anything — they say yes or no.

Progressive depth
Day 1 is a task list. The agent proposes the next rung from usage: "14 tasks mention the supplier contract — group them into a work item?" Declining costs nothing. Dormant blocks stay visible-but-quiet:

Connector blocks
Integrations are blocks: paste a key, the agent pulls the data into existing structures — GitHub PRs bind to tech items, ad campaigns to marketing items, analytics lands as evidence. Never a parallel world.

Custom agents
Describe a job → the platform drafts a named agent (a real workspace member). Then: @sourcing-scout check MOQs in any thread — it runs, reports back in-thread, artifacts arrive in the review queue.
@sourcing-scout @qa-bot

Everything generated — by meetings, agents, or automations — arrives as a proposal in one review queue: badge, provenance, accept / edit / reject. Nothing changes the workspace silently. The skeleton (ladder, loop, one-record rule) is never configurable — that's what keeps every workspace learnable for people and predictable for agents.

The whole model in one sentence

Pick a project, pick an item, see its phase, trust its why — and anything done on your behalf waits for your yes.
