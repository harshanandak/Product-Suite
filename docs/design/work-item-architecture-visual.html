00 Design principles

Graspable coreBase = task + work item + status. Everything else opt-in.
AI woven, 3 modesConfig sets up · Oversight watches · Agents launched to act.
Orthogonal axesWork carries independent refs; change focus by view, never re-parent.
Flat work spineTasks never nest; only containers nest (parentId, one policy).
Derived & totalHealth/completion/status are total functions — every case defined.
Upper = consumersStrategy + AI read the core; never complicate the base.
Viewer-independent rollupsCounts compute over the full child set; visibility hides, never counts.

03 The task and its three orthogonal axes

The heart of the model: a task carries independent references. You focus on a release or a sprint by switching the view — the task never moves.

Single-value today (one scope, one cycle) via UNIQUE(taskId) — relaxable to multi (a hotfix in two releases) additively.

04 Lifecycle — task & item

No phases. A task is a simple triad; a work item is open→done, derived-suggested and owner-confirmed. Plan/review are kinds of tasks, done anytime.

stateDiagram-v2
  direction LR
  state "TASK" as T {
    [*] --> todo
    todo --> doing
    doing --> done
    done --> [*]
  }
  state "WORK ITEM" as W {
    [*] --> open
    open --> ready_to_close : all children done (derived)
    ready_to_close --> done : owner confirms
    done --> open : child added / reopened (auto-revert)
  }
blocked is derived: a task with an incomplete dependency (e.g. execution waiting on a plan task) — that's the opt-in "gate execution when clarity is lacking." Done tasks are terminal (never blocked).

05 Union-DAG — the deadlock rule

Dependencies and containment share one acyclic graph. The rule: no task or container may depend on any of its own ancestors — this is what prevents the containment↔dependency deadlock.

graph TB
  W["Work Item W (version)"] ==> M["Milestone"] ==> T["Task T"]
  T -. "dependsOn (waits on) — OK" .-> X["Task X (sibling)"]
  T -. "depend on ancestor W — FORBIDDEN" .-> W
  linkStyle 0,1 stroke:#0e9f6e,stroke-width:2px;
  linkStyle 3 stroke:#dc2626,stroke-width:2px,stroke-dasharray:4;
Bold edges = containment (derivation, container→member). Green = a legal dependency. Red = the forbidden ancestor-dependency. Enforced transactionally server-side (concurrent local-passing writers can still form a union cycle).

06 Alignment chain (OKR)

Strategy is a separate axis. Work aligns up to an Objective; the metric lives once on the Objective's Key Results — never duplicated per work item.

graph BT
  T["Task"] -->|objectiveId / keyResultId| O
  WI["Work Item"] -->|aligns| O["Objective<br/>(owned at project / initiative)"]
  O --> KR["Key Results<br/>metric: start → current → target"]
  MS["Milestone (Alpha/Beta/GA)"] -.belongs.-> WI
  classDef obj fill:#ecfdf5,stroke:#0e9f6e,color:#065f46;
  class O,KR obj;
A task's own alignment overrides its container's (single canonical path — no double-count). KeyResult.current is the single source of truth; task completion is advisory, not an auto-metric.

07 AI — three modes + the Flag→Decision loop

AI configures the workflow, watches it, and can be launched to act. Oversight is advisory: a flag is a prompt, resolved by acting or by overriding with new reasoning — which updates the versioned decision and re-propagates alignment.

sequenceDiagram
  autonumber
  participant O as AI Oversight
  participant F as Flag
  participant U as Owner
  participant D as Decision (versioned)
  O->>F: raise (off-roadmap / cross-team conflict / comms gap)
  F-->>U: surface suggestion (never a block)
  alt accept
    U->>F: act on it → state = resolved
  else override
    U->>D: new version + new reasoning
    D->>F: state = overridden (resolvedByDecisionVersion)
    D-->>O: re-propagate new alignment
  end
Three modes: Config (AI writes the team's settings from natural language) · Oversight (continuous, emits Flags) · Agents (launched, mode = plan|review|act, over Connections, output → Attachments). All are consumers of the clean model.

09 The 8 invariants where correctness is proven

  1. Acyclic union. Dependency + containment-derivation edges form one acyclic graph; no unit depends on its own ancestor. Enforced transactionally.
  2. Axis cardinality. ≤1 scope and ≤1 cycle per task (UNIQUE on the edge tables), independent.
  3. Health is total. Leaf base cases defined; done tasks terminal; container = worst(intrinsic, worst-child); empty = "no work"; carriers = Work Item + Milestone.
  4. Close is continuous. ready-to-close (derived) + owner-confirm; attaching non-done work to a done item reverts it to open.
  5. Single alignment path. objectiveId + keyResultId consistent; task alignment overrides container's; KR.current is source of truth.
  6. Versioning. Decision & Config supersede-chains are injective, acyclic, one head; authoredBy recorded.
  7. Config default. Simplest default; absence = base experience; disabling an axis hides refs, never deletes.
  8. Referential integrity. Every ref valid or null; deletes cascade-null/block/tombstone incl. polymorphic; archived excluded from rollups.

10 Gaps — found & folded into v3 resolved

A six-perspective completeness sweep (data/backend · frontend · product · security · scale/AI-ops · domain) ran. It found three absent layers + three inconsistencies — all now added in v3 (see the layered diagram above + the architecture doc).

Verdict: the core data spine is sound (Fable-confirmed). The sweep found three whole layers largely absent — all additive to the approved shape — plus three internal inconsistencies. v3 folds every item below into the architecture. They're kept here as the record of what was added.

A · Collaboration & presentation table stakes

  • View / SavedView — the "two lenses" promise has nowhere to persist filter · groupBy · sort · layout · shared-vs-personal
  • Activity / audit stream — generalize RolloverEvent; backs the detail feed and every time-series metric (burndown, cycle-time)
  • Comments + @mentions · notifications / subscriptions / watchers
  • Task fields missing: title + description(!), priority, labels, dueDate/startDate, estimate→velocity, manual rank/order, cancelled state

B · Multi-tenancy & security substrate

  • Tenant / Org boundary above Team + a tenant-isolation invariant (every ref same-tenant — today a ref can cross tenants)
  • Connection credentials — encrypted secrets, scopes, refresh, revocation (agents act through these)
  • Membership + roles / RBAC (edit/close/override/configure); PII / right-to-erasure; SSO / SCIM; guest sharing

C · AI / async runtime makes it real

  • Jobs / scheduler tier — rollover, auto-create, continuous Oversight, sync, notifications are all background work with no home
  • Agent definition (registry) vs AgentRun lifecycle (state · cost · authority) + act-mode guardrails (dry-run→approve · budgets · kill switch)
  • Connection sync — webhook/poll · cursor · ExternalRef mapping (which PR ↔ which task); Flag dedup/severity/audience
Fixed in v3 — internal inconsistencies (real, not just omissions):
  • Invariant 3 references a Task dueDate that doesn't exist — leaf at_risk = overdue has no field to read. Add dueDate, or derive at_risk from container dates only.
  • completion% — count-based vs effort-weighted is asserted but never specified (needs the estimate field to settle).
  • No cancelled / won't-do terminal — invariants exclude "terminal" tasks though only done exists; abandoned work pollutes sprints / health / completion%.
Coverage caveat: the data-backend lens failed on API rate-limiting — indexes, transactions, query patterns, and the realtime contract are only partially covered (by the scale/domain lenses). Re-runnable on request.
